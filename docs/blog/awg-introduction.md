# Your Codebase Has Amnesia — Sensei Is the Cure

Every codebase has rules that aren't in the code.

"Don't use `os.Getenv()` for config — it must come from the config store." "The install path and the CLI path must both write the checksum sidecar." "Never call `pkill -f` — it matches the parent shell and self-kills."

These rules live in someone's head, in a post-mortem nobody reads, in a Slack thread from six months ago. They are load-bearing knowledge — violate them and something breaks. But they are invisible to the next person who touches that code.

When that next person is an AI agent, the problem gets worse. AI agents are confident, fast, and have no institutional memory. They will make the "obvious" fix that a senior engineer would never make, because the senior engineer remembers the last time someone tried that and it took down production.

We built Sensei to fix this.

---

## What Sensei does

Sensei makes architectural intent queryable at the point of edit. Before you change a file, Sensei tells you what you need to know:

```
$ sensei briefing --file src/auth/session.go

Status: OK

Invariants:
  - [critical] auth.session_token_must_be_httponly
    "Session tokens must use HttpOnly cookies — never accessible to JavaScript"

Forbidden fixes:
  - expose_token_in_response_body_for_spa
  - add_javascript_readable_token_cookie

Required tests:
  - TestSessionCookieIsHttpOnly

Failure modes:
  - [critical] auth.xss_steals_session_token
    "XSS attack steals session token from non-HttpOnly cookie"
```

This isn't generated by an LLM. It's compiled from YAML files that your team writes — encoding real incidents, real invariants, real forbidden fixes. The graph is deterministic: same input, same output, every time.

## Where it came from

Sensei was born inside [Globular](https://github.com/globulario), a 465,000-line distributed platform with 33 services. We build Globular with AI agents (Claude, GPT, Codex) doing significant portions of the implementation work. The agents are productive — but they kept introducing the same class of bugs.

Not syntax errors. Not logic bugs a linter would catch. Architecture violations.

A function wrapper that looked mechanical but bypassed a manifest verification check. A "convenient" fallback that returned the same shape as the real response, making it impossible for callers to distinguish truth from guess. A heartbeat handler that wrote desired-state fields it was only supposed to observe.

Each of these shipped, broke something, required a patch release, and got a post-mortem. But post-mortems don't prevent recurrence — they document what happened, not what should happen next time. The same agent, in a new session, with no memory of the last incident, would make the same mistake.

We needed something that would intercept the edit before it happened and say: "This file is protected by invariant X. Here's what broke last time someone did what you're about to do."

## How it works

Sensei is three things:

**1. A set of YAML files you write.**

You encode your project's architectural knowledge as invariants (rules that must hold), failure modes (things that broke or could break), and incident patterns (edit shapes that introduce bugs):

```yaml
invariants:
  - id: config.no_env_vars_for_service_config
    title: Service config must come from the config store, not environment variables
    severity: critical
    protects:
      files: [src/config/loader.go]
    forbidden_fixes:
      - add_env_var_fallback_for_convenience
    required_tests:
      - TestConfigLoaderRejectsEnvVars
    related_failure_modes:
      - config.env_var_overrides_production_setting
```

**2. A knowledge graph compiled from those YAML files.**

`sensei build` compiles your YAML into RDF triples and loads them into Oxigraph, a lightweight SPARQL store. The graph connects files to invariants, invariants to failure modes, failure modes to forbidden fixes. When you query "what do I need to know about this file?", the graph traverses these connections and returns everything relevant.

**3. A query layer that fires before edits.**

`sensei briefing`, `sensei impact`, and `sensei preflight` are thin gRPC clients that query the graph. Claude Code hooks can enforce consultation before edits — if you try to modify a file in a protected directory without calling briefing first, the hook blocks the edit.

## What makes this different

There are linters, architecture tests, ADRs, and post-mortems. Sensei is different in three ways.

**It's queryable at the point of edit, not in a document somewhere.**

An ADR that says "we use HttpOnly cookies" is useful when someone reads it. It's useless when someone is editing `session.go` and doesn't know the ADR exists. Sensei surfaces the relevant knowledge exactly when and where it's needed — before the edit is made.

**It encodes forbidden fixes, not just rules.**

The most valuable part of an invariant is often what NOT to do. "Don't add an env var fallback for convenience" is more actionable than "use the config store." Forbidden fixes catch the "obvious" solutions that are known-broken — the ones that look correct to someone who doesn't know the history.

**It compounds with every incident.**

Every bug you encode makes the graph more valuable. After 50 incidents, the graph contains a dense map of your codebase's danger zones. New developers (and AI agents) get the benefit of that institutional memory from their first edit.

## The meta-principles

While encoding 50+ incidents in Globular, we noticed they kept falling into the same patterns. A fallback that hides a failure. A write with no cleanup path. Two writers racing on the same state. An intermediate state that looks done.

We distilled these into 134 meta-principles organized in eight categories:

**Authority** (20 principles) — "Who owns this truth, and is this code that owner?"

When storage is accessed directly instead of through its owning service. When the same field means different things depending on who wrote it. When two writers compete without convergence or fencing.

**Signal** (19 principles) — "Is the truth arriving intact?"

When a fallback returns the same shape as the real response. When "not found here" is treated as "doesn't exist anywhere." When a TLS error is absorbed into a generic timeout.

**Lifecycle** (38 principles) — "Will this operation complete?"

When a write creates a pending record with no cleanup path. When an intermediate state satisfies a completeness check. When a retry loop amplifies a failure instead of contracting it.

**Dependency** (7 principles) — "What breaks if something non-critical fails?"

When a heartbeat handler calls an AI service and the AI service being slow causes nodes to be marked offline. When a deployment system can't deploy the fix for itself.

**Perception** (19 principles) — "Is the UI telling the truth about system state?"

When a green badge reports stale runtime state. When a warning is only visible in color (invisible to colorblind users and screen readers). When a dashboard metric is cached.

**Composition** (7 principles) — "Does the layout make truth perceivable?"

When a decorative card is louder than a drift warning. When a success color is shown on an unconfirmed state.

**Structure** (12 principles) — "Is the code shaped to last?"

When a generic mega-table is merged on resemblance. When a pass-through wrapper hides nothing and adds confusion.

**Evolution** (11 principles) — "How may the project change safely over time?"

When a refactor ships without a test pin. When a branch can't be reviewed in slices. When a build is non-deterministic.

These principles are universal — they apply to any software system, not just distributed infrastructure. Sensei ships them as seed content with every new project.

The real power is using them as a lens after an incident. "This bug was a `meta.fallback_must_degrade_semantics` violation. Where else in our codebase do we have fallback paths that return the same shape as the primary?" The principle tells you where the next bug is hiding, in code that hasn't broken yet.

## The cold-start problem

An empty awareness graph is useless. This is the adoption challenge: the tool is only valuable once you've encoded enough knowledge, but encoding knowledge takes effort.

We solve this three ways:

**Start with your most painful recent bugs.** Don't try to encode everything. Take the last 3-5 bugs that required patch releases or caused outages. Each one becomes an invariant + failure mode + incident pattern. Five entries is enough to start getting value.

**Ship the meta-principles as seed content.** `sensei init` creates `meta_principles.yaml` with all 134 principles. They're immediately queryable. When your first bug doesn't match any of your project-specific invariants, it probably matches a meta-principle — and that principle tells you what to search for.

**Make the incident-to-invariant workflow fast.** After fixing a bug, encoding it should take 5 minutes: add the invariant, add the failure mode, run `sensei build`. If it takes longer, people won't do it. The YAML schema is designed to be writable without reading documentation.

## AI agents and drift prevention

The original motivation for Sensei was preventing AI agents from introducing architecture violations. But the mechanism is more general — it prevents drift.

Architecture drifts one "simple fix" at a time. No single commit looks wrong. The damage is cumulative, invisible in isolation, and only becomes apparent when the system starts failing in ways that are expensive to trace. By the time you notice the drift, reversing it requires understanding the intent that was never recorded.

Sensei records the intent. The briefing connects a local code change to the global architectural picture. It doesn't replace code review — it makes code review more effective by surfacing the context that reviewers would otherwise need to hold in their heads.

For AI agents specifically, Sensei closes the memory gap. An agent starts every session from zero. CLAUDE.md helps, but it's prose — agents skim it the way humans skim documentation. Sensei is structured, queryable, and fires at the exact moment the agent needs it. The hook enforcement means the agent can't skip it, even when it's "confident" the fix is simple.

The cases where you are most confident a fix is "too simple to need awareness" are exactly the cases where you are most likely to miss a critical invariant. We learned this the hard way — a one-function wrapper that looked mechanical shipped without manifest verification because it seemed too obvious to check. Sensei exists so that "too obvious" is not an exemption.

## Getting started

```bash
# Install (builds sensei + server, fetches oxigraph — no Docker needed)
git clone https://github.com/globulario/sensei.git
cd sensei && ./scripts/install.sh
export PATH="$PWD/bin:$PATH"

# Scaffold your project
cd /path/to/your-project
sensei init

# Start the graph and compile your awareness YAML
sensei serve -no-seed &
sensei build

# Query it
sensei briefing --file <your-critical-file>
```

The [quickstart guide](https://github.com/globulario/sensei/blob/master/docs/quickstart.md) walks through the full setup in 10 minutes.

The [meta-principles reference](https://github.com/globulario/sensei/blob/master/docs/meta-principles.md) documents all 134 principles with examples and search patterns.
